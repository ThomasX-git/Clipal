## v0.11.3

### English

#### Highlights

- Fixed unified-ingress compatibility so upgraded clients that send official bare API paths under `/clipal` keep working instead of failing at protocol detection.
- Made OpenAI Responses routing under `/clipal` understand official subpaths, so response-resource operations stay on the correct routing scope.

#### What Changed

- Fixed: `/clipal` now canonicalizes supported official bare paths before protocol-family detection instead of requiring every client to send the fully versioned canonical path
- Fixed: bare OpenAI-compatible paths such as `/clipal/responses`, `/clipal/models`, and other supported `/clipal/*` OpenAI resource paths now normalize to `/clipal/v1/*`
- Fixed: bare Claude-compatible paths such as `/clipal/messages` and `/clipal/messages/count_tokens` now normalize to `/clipal/v1/*`
- Fixed: bare Gemini RPC paths such as `/clipal/models/{model}:generateContent` now normalize to `/clipal/v1beta/*`
- Fixed: OpenAI Responses official subpaths under `/clipal`, including resource and action routes, now stay classified as `openai_responses` instead of being rejected as unknown endpoints
- Improved: unified-ingress routing tests now cover canonicalization of bare official paths and mixed OpenAI/Gemini model-route disambiguation

### 中文

#### 亮点

- 修复了统一入口 `/clipal` 的兼容性问题，升级后的客户端如果发送官方风格的裸路径，不会再因为协议识别过严而直接失败。
- `/clipal` 下的 OpenAI Responses 现在能正确识别官方子路径，response 资源相关操作会继续落在正确的路由 scope 上。

#### 变更内容

- 修复：`/clipal` 现在会在协议族识别前，先把支持的官方裸路径规范化，而不是要求所有客户端都发送完全 versioned 的 canonical path
- 修复：像 `/clipal/responses`、`/clipal/models` 以及其他受支持的 OpenAI 风格 `/clipal/*` 资源路径，现在会自动归一化为 `/clipal/v1/*`
- 修复：像 `/clipal/messages`、`/clipal/messages/count_tokens` 这样的 Claude 风格裸路径，现在会自动归一化为 `/clipal/v1/*`
- 修复：像 `/clipal/models/{model}:generateContent` 这样的 Gemini RPC 裸路径，现在会自动归一化为 `/clipal/v1beta/*`
- 修复：`/clipal` 下 OpenAI Responses 的官方子路径，包括资源路径和动作路径，现在会继续被识别为 `openai_responses`，不再被误判为 unknown endpoint
- 改进：统一入口路由测试现在覆盖官方裸路径的规范化，以及 OpenAI/Gemini 模型路径混合场景下的歧义分流
